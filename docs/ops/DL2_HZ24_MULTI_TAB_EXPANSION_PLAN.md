# DL2 HZ24 多 Tab 商品池扩展计划

更新日期：2026-06-16

## 目标

评估“商品推广”页面中除“全部商品”外的其他 Tab 是否包含显著增量 SKU，并在不破坏 HZ23 商用基线的前提下扩充佣金商品池。

## 阶段 1：只读重叠审计

使用现有授权会话串行点击各 Tab，只读取：

- Tab 名称；
- 页面展示总量；
- 首屏 SKU；
- 与“全部商品”首屏的交集、独有 SKU 和 Jaccard；
- 风控信号。

该阶段不点击“一键领链”，不写入 HZ23 catalog，不改变当前 3304 条候选。

## 阶段 2：完整 SKU 审计

仅对首屏出现明显增量的 Tab 执行完整分页扫描。每个 Tab 使用独立命名空间：

- `data/state/tabs/<tab_id>/`
- `reports/tabs/<tab_id>/`
- `data/import/tabs/<tab_id>/`

所有 Tab 扫描结果按 SKU 全局去重，记录 `source_tabs` 和首次发现 Tab。

## 阶段 3：推广链接生成

只对“相对当前可信商品池的增量 SKU”生成推广链接，禁止对重叠 SKU 重复领链。

推广链接必须带有生成账号/结算主体 lineage。不同结算主体生成的推广链接不得混合发布。正式发布默认只使用公司指定的 canonical settlement identity。

## 并行策略

如果只有一个京东联盟授权账号：

- 使用单一浏览器身份串行采集多个 Tab；
- 不复制 Cookie 并发；
- 不同时运行多个领链任务。

如果具备四个由平台允许、属于同一业务主体的独立授权身份：

- 每个身份使用独立 OS 用户或独立 Chrome profile；
- 使用不同 CDP 端口；
- Cookie、Profile、state 和日志完全隔离；
- 每个 worker 只负责一个 Tab；
- 调度错峰，统一由中央 merger 按 SKU 去重；
- 遇到真实验证立即停止，不绕过。

## 与当前商用基线的关系

HZ23 的 3304 条候选先完成 MySQL 回填和连续七天稳定期。HZ24 新增 Tab 数据先作为隔离候选验证，不改变稳定期起始基线。增量池通过完整轮次、链接可信、唯一 SKU、checksum/hash 和回滚测试后，以新的 publish version 合并发布。
